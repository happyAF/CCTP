Chapter 01

이 프로젝트의 무게중심은 어디에 있는가

"음악 같이 듣기"는 표면이다. 메인 디시는 수평 확장 가능한 실시간 시스템을 클라우드 위에 올리는 일.

음악 재생 자체는 클라우드 컴퓨팅과 직접적 관련이 없다. 단일 서버 웹 앱으로도 충분히 구현 가능하다. 평가 포인트는 그 다음에서 나온다 — 사용자가 늘었을 때 어떻게 서버를 늘리고, 늘어난 서버들이 어떻게 같은 방의 상태를 공유하며, 미디어 파일을 어디에 저장하고, 부하에 어떻게 자동 대응하는가.

North Star 발표/보고서의 메인 스토리는 "음악 서비스 만들었어요"가 아니라 "실시간 시스템을 어떻게 수평 확장했나"이다.
평가 포인트 왜 클라우드 컴퓨팅인가
Multi-EC2 + LB 단일 서버는 트래픽·장애에 무력하다. 수평 확장이 클라우드의 출발점.
Redis Pub/Sub WebSocket은 stateful — 서버가 둘 이상이면 같은 방의 메시지가 끊긴다. 서버 간 중계가 필수.
S3 Storage EC2 로컬 디스크에 저장하면 다른 서버에선 보이지 않는다. 공유 스토리지가 답.
Docker 어디서나 동일 환경. Auto-scaling의 전제 조건.
Auto-scaling 부하에 따라 자동 증감. 클라우드가 약속한 핵심 가치.

Chapter 04

동기화는 이렇게 흐른다

단순함이 안정성이다. MSE도, fragmented MP4도 쓰지 않는다. 브라우저 기본 <audio> 태그만으로 충분하다.

  • 누군가 mp3를 업로드한다 → 서버가 S3에 저장하고 URL을 받는다.
  • 서버는 모든 클라이언트에게 메시지를 보낸다 — { url, play_at: now + 3s }.
  • 각 클라이언트는 mp3를 미리 다운로드한다 (preload).
  • 다운로드 완료 → 서버로 ready 신호 전송.
  • 전원 ready가 되면 서버가 최종 play_at 시각을 확정 → 모두 동시에 재생.
  • 늦게 합류한 사람은 { url, seek_to: N }으로 위치 맞춰서 재생.
Trade-off 클라이언트별 다운로드 속도 차이로 첫 재생에 0.5~1초 오차가 생길 수 있다. ready 체크 로직이 이를 완화한다. 완벽한 ms 동기화는 목표가 아니다 — "다 같이 듣고 있다"는 체감이 목표다.

Chapter 07

예상 리스크

RISK 01

AWS 비용

EC2 여러 대 + LB + Redis 구성은 프리티어를 초과할 수 있다. 개발은 로컬 Docker에서, AWS 배포는 발표 직전 며칠로 압축한다.

RISK 02

WebSocket + LB 조합의 함정

Sticky session 설정이 누락되면 같은 방 유저가 다른 서버에 붙는다. 이 구성에서 며칠을 잡힐 수 있다. 처음부터 가정에 넣고 출발한다.

RISK 03

동기화 정확도의 한계

네트워크 환경에 따라 첫 재생에 오차가 발생한다. "완벽한 ms 싱크"가 목표가 아니라 "체감상 같이 듣는 느낌"이 목표라는 점을 발표에서 명시한다.

Chapter 10

팀이 굴러가는 법

도구. GitHub (코드 + Issues), Discord 또는 카톡 (일상), Notion 또는 Google Docs (문서), Figma 또는 그림판 (와이어프레임).

브랜치. main = 발표용 안정 / dev = 통합 / 각자 feature/이름-기능 브랜치 → dev로 PR.

미팅. 주 2회, 30분~1시간. 진행상황 / 막힌 부분 / 다음 주 계획. 회의록은 Notion에 남긴다.

의사결정. 스택·스코프 변경 = 3명 모두 동의. 작은 디테일 = 담당자 결정 + 채널 공유.

24시간 룰 혼자 24시간 이상 막혀 있지 않는다. 학부 팀플이 망하는 가장 흔한 패턴은 한 명이 막혀서 침묵하는 것이다. 막히면 말한다 — 자존심보다 마감이 가깝다.
Chapter 11

GitHub은 이렇게 쓴다

처음에 한 번 정해두지 않으면 main 브랜치가 박살난다. 이 챕터는 우리 팀 규칙이다.

레포 주소. github.com/happyAF/CCTP

브랜치 구조

브랜치 용도
main발표용 안정 버전. 직접 push 금지. dev에서 PR로만 머지.
dev통합 작업 브랜치. 각자 feature 브랜치에서 PR로 머지.
feature/이름-내용각자 작업 브랜치. 예: feature/wonjae-upload-ui

폴더 규칙

각자 자기 트랙 폴더 안에서만 작업한다. 다른 트랙 폴더를 건드려야 하면 미리 디코에 공유.

frontend/Track A — 웹 UI, 클라이언트 로직
backend/Track B — WebSocket, API
infra/Track C — Docker, AWS, 배포 스크립트
docs/공동 — 제안서, 회의록, 다이어그램

작업 흐름

  • 최신 dev 받기
    git checkout dev && git pull
  • feature 브랜치 따기
    git checkout -b feature/내이름-작업내용
  • 작업 + 커밋 — 한 기능 단위로 자주 커밋. "완성"까지 기다리지 말기.
  • push
    git push -u origin feature/내이름-작업내용
  • GitHub에서 Pull Request 생성 → 대상은 dev 브랜치
  • 다른 팀원 1명 리뷰 → 승인되면 머지. 본인이 직접 머지 X.

커밋 메시지

한국어든 영어든 상관없지만 "뭘 했는지"는 명확히. "수정", "업데이트" 같은 거 금지.

Good 방 생성 API 추가
fix: WebSocket 재연결 시 방 목록 안 나오던 버그
infra: docker-compose에 redis 추가

Bad 수정 · 업데이트 · ㅁㄴㅇㄹ

PR 규칙

  • 제목에 트랙 prefix — [FE] / [BE] / [INFRA]
  • 본문에 "뭘 했는지 + 어떻게 테스트했는지" 3~5줄로
  • 리뷰는 24시간 안에. 자기 트랙 PR이 아니어도 한 번 훑어보기.
  • 충돌나면 본인이 해결 — dev를 자기 브랜치로 머지하거나 rebase 후 다시 push.

Issues 활용

할 일 / 버그 / 아이디어는 GitHub Issues에 기록한다. 디코에서만 얘기하고 끝내지 말기 — 회의 끝나면 사라진다.

  • 제목에 트랙 prefix — [FE], [BE], [INFRA]
  • 본인 작업은 본인한테 Assign
  • PR 본문에 Closes #12 식으로 적으면 머지될 때 이슈 자동 닫힘

올리지 말 것

절대 커밋 금지 AWS 키 · 비밀번호 · API 토큰 · .env 파일 · *.pem. 한 번 push되면 GitHub 히스토리에 영원히 남는다. 한 번이라도 실수했으면 즉시 채널에 공유 + 키 폐기.

비밀값은 .env에 두고 .env.example만 올린다 (값 비운 채로). 실제 값은 디코 비밀 채널이나 별도 채널로 공유.

  • main에서 Release 태그 만들기 (예: final)
